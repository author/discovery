Microservice Discovery

The Challenge:

Microservices architecture requires the components within the environment to communicate with each other. This implies each service knows how to contact the others. This seemingly simple task becomes more challenging when the makeup of the environment changes, such as adding/removing/updating services.

dicovery.author.io acts as a catalog of microservices. Each microservice can obtain an updated list of other services within it's network. Using the REST API, managing networks and nodes (services) is a trivial task.

For example, a microservice environment may consist of three separate services running indepedently, possibly even on servers around the world. The discovery service provides a single place to retrieve microservice data.

Microservice environments are often dynamic. The catalog can change regularly. To support this, the discovery service has a webhook event system. Using this system, it is possible for microservice environments to react to changes in the network as they happen.

Getting Started

  1. First, there are special instructions for gaining access to the API. You can activate these instructions by reading and agreeing to the terms of service and privacy policy (sorry, we have to require this).
  2. Create a network. (see how to create a network)
  3. Start adding nodes (services). (see how to add a node)
  4. Check out the webhooks documentation to react to changes in your new network catalog.

Why was this Created?

We've been working on a microservices platform called NGN on and off over the last few years. With new tools like Docker, Node.js, and Go, the case for microservice infrastructure has become clearer. At Author.io, we are obsessed with making great developer experiences. We felt this discovery service was a simple way to enhance the experience of microservice creation.

More specifically, we're using this service with NGN. NGN has a container called the "NGN Bridge". It acts like a traffic cop, directing traffic between microservices. The discovery service allows multiple NGN Bridges to communicate with each other, even if they're not on the same local area network. As a result, NGN can communicate with just about any microservice, whether it's on a LAN/WAN/Cloud/etc.
